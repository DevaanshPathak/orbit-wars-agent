# v17 Notes

v17 trains an 8-member ensemble ranker on top of the **v16 heuristic** using a **new dataset generated by the v16 agent** (auto-discover newest `data/*/candidates_v7.csv` from HF). Key improvements over v15: fixed training labels (win/loss/tie branches) and larger model (256→128→64).

## Files

- `train_v17_ranker.py` — v17 trainer; auto-discovers newest HF `data/*/candidates_v7.csv`; artifacts to `v17/` on HF.
- `v17_training_policy.ipynb` — self-contained Kaggle/Colab notebook; asks for `HF_TOKEN`, auto-discovers newest HF dataset (shows all available), uploads to `devaanshpa/orbit-wars-agent/v17`.
- `notes.md` — this file.

## Dataset

Generate a new dataset using the v16 heuristic before training:

```bash
python generate_training_data.py --games 5000 --both-sides --workers 16 --max-candidates-per-turn 64
```

This produces `data/<timestamp>/candidates_v7.csv` with the fixed label logic (win/loss/tie branches, lowered counterfactual thresholds). Upload via `--upload` flag or manually after generation. The notebook auto-discovers the newest timestamp folder on HF.

Expected stats vs v15 dataset:
- `positive_rate` ~35% (was ~70% — fixed by win/loss label split)
- `counterfactual_rate` significantly higher (thresholds lowered: rank ≤ 4, gap ≥ -0.15, delta_15 < -5.0)

## Heuristic Baseline

v16 heuristic (see `notebooks/v16/notes.md`):
- Attack-timing bonus when enemy fleet departs target
- ETA filter in multi-source bundles (drops sources with `src_eta > target_turn + 3`)
- Neutral-denial bonus when leading production by 10%+ after step 60
- Late-endgame reserve strip at step 380+ with 35%+ ship lead
- Staging guard: aborts if enemy fleet arrives at front within 12 turns

Plus all v13 features:
- Staging re-enabled with contested-front guard (enemy ETA ≤ 15)
- Beam width: 5
- Per-planet production-race drawdown (safe planets only, enemy ETA > 20)
- Opportunity scoring (+production × 8.0)
- Comet ETA cap 22 when leading production

## Model Architecture

256 → 128 → 64 → 1 (ReLU activations, Dropout 0.12/0.084/0.054)

Increased from v15's 160 → 80 → 40 → 1 for better capacity on the larger dataset and richer label signal.

## Training Defaults

Slightly tuned from v15 for the cleaner label distribution:

- Ensemble size: 8
- Epochs: 300 (increased from 280 — cleaner labels converge more slowly)
- Batch size: 4096
- Pair loss weight: 1.10 (increased from 1.05 — stronger pairwise signal on clean negatives)
- Max pairs per turn: 14 (increased from 12)
- Learning rate: 0.00045
- Weight decay: 0.00030
- Dropout: 0.12
- Patience: 40 (increased from 36)
- Score scale: 205
- Checkpoint every: 40 epochs → `v17/checkpoints/` on HF

## Run Order

1. Generate new training data with v16 heuristic (see above). Wait for upload to HF.
2. Open `v17_training_policy.ipynb` on Kaggle (GPU T4 x2) or Colab.
3. Enter `HF_TOKEN` when prompted (cell 1).
4. Run all cells. Cell 3 auto-discovers and shows all available datasets, picks newest.
5. Checkpoints upload every 40 epochs; final artifacts go to `v17/`.
6. Download `model_weights_v17.json` to local `models/`.
7. Build submission:
   ```bash
   python build_submission.py --weights models/model_weights_v17.json --output models/v17_kaggle/main.py
   ```
8. Submit:
   ```bash
   kaggle competitions submit orbit-wars -f models/v17_kaggle/main.py -m "v17 v16 heuristic + ensemble on new dataset (fixed labels)"
   ```

## Why the Label Fix Matters

In v15, `positive_rate = 0.696`: 70% of all candidate rows were labeled positive because `label = 0.48 + 0.30 * delta_signal` gave >0.5 for most winning-game moves. The model had almost no negatives to learn from, and `counterfactual_rate = 0.00094` provided nearly no "better alternative" signal.

v17 training data (generated with `generate_training_data.py` after the v16 commit) uses:

| Outcome | Label formula | Expected range |
|---------|--------------|----------------|
| Win (`result > 0`) | `0.53 + 0.24 * delta + 0.08 * margin` | 0.30 – 0.65 |
| Loss (`result < 0`) | `0.30 + 0.14 * delta` | 0.16 – 0.44 |
| Tie | `0.42 + 0.10 * delta` | 0.32 – 0.52 |

Expected `positive_rate` ~35%, giving the model real win vs. lose discrimination.
